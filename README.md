# IT-230

# Summary of the Application and its Goals
The application was designed to help a user register for courses while enforcing specific business rules, such as preventing duplicate course selections and limiting total credit hours. The console version focused on basic input, validation, and output, while the WPF version expanded those same requirements into a graphical interface that allowed users to select courses from dropdown lists and see their registered courses displayed on screen. Across both versions, the goal was to create a simple, reliable tool that guided the user through the registration process without errors or confusion.

# What Went Particularly Well
The strongest part of the development process was translating the requirements into clean, organized code. Creating the "Course" class, managing course objects, and implementing validation logic all went smoothly. In the WPF version, wiring up event handlers and ensuring the UI responded correctly to user actions was another area that came together well. The final application behaved consistently, handled invalid input gracefully, and produced clear output, which reflects careful attention to detail and a solid understanding of object‑oriented design.

# Comparing the Console and WPF designs
The console application required only a linear flow: prompt the user, read input, validate it, and display results. The WPF application required a more thoughtful, user‑centered design. Key features include: A main window with ComboBoxes for course selection, Buttons to add courses and finalize registration, a ListBox showing the user's selected courses, and Error messages and validation feedback displayed in real time. Designing the WPF UI meant thinking about how users naturally interact with software—minimizing typing, preventing invalid actions, and making the interface visually clear. The layout, control choices, and event handling all supported a smoother experience than the console version. The design succeeded because it reduced user effort and made the rules of the system visible and intuitive.

# Debugging and coding approach
Debugging began with stabilizing the existing code: checking namespaces, verifying constructors, confirming event wiring, and ensuring the UI elements matched the code‑behind. Techniques that were especially helpful included: Reading compiler errors carefully and tracing them back to the source, Testing frequently in small increments, Using Visual Studio’s error list, IntelliSense, and build tools, as well as Printing or displaying intermediate values to confirm logic. These strategies made it easier to isolate issues and confirm that each part of the program behaved as expected. They are the same techniques that will be useful in any future project—especially when working with unfamiliar code or larger applications.

# Innovation and overcoming challenges
One of the biggest challenges was translating console‑based logic into a WPF environment. The shift required rethinking how data flows through the program: instead of reading input line by line, the UI needed to respond to user actions through events. Managing state, validating selections, and updating the interface dynamically required a more interactive mindset. Overcoming this meant experimenting with event handlers, testing different UI layouts, and ensuring the logic stayed consistent across both versions of the application. This process encouraged creative problem‑solving and a deeper understanding of how user interfaces and business logic work together.
